Attacks on encryption

| Attack                      | Attacker access                                                                         |
| --------------------------- | --------------------------------------------------------------------------------------- |
| **Ciphertext only attack**  | attacker only has the ciphertext                                                        |
| **Known plaintext attack**  | attacker has a collection of plaintexts and their corresponding ciphertexts             |
| **Chosen plaintext attack** | attacker has access to an encryption oracle<br>can get ciphertext for **any** plaintext |

### Concept
#### Security requirements
Confidentiality
- only the sender and receiver can **understand** the message contents
- any eavesdropper should not be able to decipher any useful information form the ciphertext

Integrity
- ensure that the message was **not altered** in transit, **without detection**

Authenticity
- **confirm the identity** of the other party

Non-repudiation
- prove to a 3rd party
- sender cannot deny that they created the message

Malicious actions
- eavesdrop - intercept messages
- injection - inserting messages into a connection
- spoofing/impersonation - fake source address
- hijacking - take over an ongoing connection, replace either party or become man-in-the-middle
- [denial of service](/labyrinth/notes/cs/cs2107/denial_of_service) - prevent others from using a service
#### [Symmetric key encryption](/labyrinth/notes/cs/cs2107/encryption#^c38796)
- encryption and decryption keys are the same

$$
m = K_{S}(K_{S}(m))
$$

- eg.
	- ceasar's cipher - [monoalphabetic substitution cipher](/labyrinth/notes/cs/cs2107/substitution_ciphers)
	- polyalphabetic encryption w cycling pattern
	- DES, AES - [block ciphers](/labyrinth/notes/cs/cs2107/block_ciphers)
- weakness:
	- sender and receiver need to know a shared key
	- how to agree on the key securely

#### [Public key cryptography](/labyrinth/notes/cs/cs2107/public_key_cryptography)
- encryption and decryption keys are **not related**
- anyone can encrypt with public key, only receiver can decrypt with private key

$$
m = K_{B}^-(K_{B}^+(m))
$$

- eg.
	- diffie-hellman
	- [RSA](/labyrinth/notes/cs/cs2107/textbook_RSA)

Session keys
- use PKC to estabilish a symmetric key
- PKC is usually expensive, symmetric key is cheaper

#### [Hashing](/labyrinth/notes/cs/cs2107/cryptographic_hashing)
- one-way, no unhashing
- designed to prevent collisions

$$
\text{Hard to find: }x,y \text{ such that }H(x) = H(y)
$$
> unlike [CRC](/labyrinth/notes/cs/cs2105/CRC) where it is possible to find collisions

- uses:
	- software integrity - check downloaded binaries
	- detect virus - without providing the virus itself
	- password hashing - to avoid storing the actual password
	- timestamp/proof of work - ensure nothing is changed
	- data integrity - blockchain

$$
\text{Verify: }H(m) = H(m') \implies m = m'
$$

#### Message authentication code([MAC](/labyrinth/notes/cs/cs2107/authentication#^29278f))
- hasing with a shared key

#### Digital signature
- sender signs a message, establishing owenership
- sender encrypts using private key, anyone can verify using public key
- verifiable - ability to check that the signature and message was generated by the sender
- unforgeable - only the sender could have generated the signature and message
- MAC but with non-repudiation, only sender has the private key

$$
m = K_{B}^+(K_{B}^-(m))
$$
> requires the homomorphic property of RSA

Public key infrastructure([PKI](/labyrinth/notes/cs/cs2107/PKI))
- need to make sure the public key is public/distributed

Certification autority(CA)
- issues and signs digital certificates to websites/entities
- ensure that its own public key is securly distributed
- authenticate the website's public key

$$
\begin{align*}
A\text{ sends message:}&&& {\color{red} m || K_{A}^-(H(m)) }\\
A\text{ sends signed public key:}&&& {\color{red} K_{A}^+||K_{CA}^-(H(K_{A}^+)) }\\
\\
B\text{ verifies until root CA}&&&\dots\\
\\
B\text{ knows that }K_{A}^+\text{ is trusted:}&&& {\color{cyan} K_{A}^+ }\\
B\text{ decrypts the hash:}&&& {\color{cyan} H(m) }= {\color{cyan} K_{A}^+({\color{red} K_{A}^-(H(m)) }) }\\
B\text{ verifies the message:}&&& \text{check }{\color{cyan} H(m) } = H({\color{red} m }) 
\end{align*}
$$

#### Secure email
- combination of mechanisms to ensure secure communication

$$
\begin{align*}
A\text{ has message:}&&& m\\
A\text{ generates symmetric key:}&&& K_{S}\\
\\
A\text{ digitally signs }m:&&& K_{A}^-(H(m))\\
A\text{ encrypts using the symmetric key:}&&& K_{S}(m||K_{A}^-(H(m)))\\
\\
A\text{ encrypts the symmetric key for }B:&&& K_{B}^+(K_{S})\\
\\
A\text{ sends:}&&& {\color{red} K_{S}(m||K_{A}^-(H(m))) || K_{B}^+(K_{S}) } & \text{Potentially modified}\\
\\
B\text{ decrypts the symmetric key:}&&& {\color{cyan} K_{S} }=K_{B}^-({\color{red} K_{B}^+(K_{S}) })\\
\\
B\text{ decrypts the message:}&&& {\color{cyan} m||K_{A}^-(H(m)) } = {\color{cyan} K_{S}( {\color{red} K_{S}(m||K_{A}^-(H(m))) }) }\\
B\text{ decrypts the signature:}&&& {\color{cyan} H(m) } = K_{A}^+({\color{cyan} K_{A}^-(H(m)) })\\
\\
B\text{ verifies the message:}&&& \text{check }{\color{cyan} H(m) } = H({\color{cyan} m }) 
\end{align*}
$$
